This page presents results from a Studienarbeit investigating whether a complete keyword spotting pipeline — training on consumer GPU, deployment on ARM — can be built with standard PyTorch only.

Four neural architectures (CNN, ResNet, DS-CNN, BC-ResNet) were trained on the full 36-class Google Speech Commands dataset and benchmarked on two Raspberry Pi boards across five independent runs each.

Key Findings
The central conclusions from the Studienarbeit.
📐
Parameter–Latency Paradox
BC-ResNet has only 30k parameters (vs. 112k for ResNet) yet runs 3.3× slower on Pi Zero. Reason: 43 sequential layers create per-layer dispatch overhead in PyTorch's eager mode that dwarfs actual compute time.
⚡
INT8 Null Effect
Dynamic INT8 quantization delivers no speed gains on any ARM platform. quantize_dynamic only touches Linear layers — the conv-dominated models stay FP32 where it counts. For the few quantized tensors (0.12–0.71 MB total) the per-pass dequantization overhead eats any bandwidth advantage. The apparent ResNet tail-suppression on Pi Zero does not hold up — it is run-dependent noise, not a real effect.
🌡️
Thermal Stability
On Pi Zero 2W, DS-CNN (81.1°C) and BC-ResNet (81.7°C) both cross the 80°C soft-throttle threshold under INT8, and CNN reaches 80.6°C. ResNet stays clear at 72.5°C (FP32) / 75.8°C (INT8) — the only thermally safe, latency-stable choice, and the reason FP32 ResNet is the recommendation here.
🎯
Platform Transfer Effect
On Pi 4B, all four models comfortably meet the 500ms real-time requirement, making architecture choice secondary. The ranking inverts on Pi Zero 2W: only layer count — not parameter count or MACs — predicts latency stability.
📊
Accuracy Convergence
CNN, ResNet, DS-CNN and BC-ResNet all converge to the same accuracy tier (~94–95%) on the harder 36-class task. Only CNN falls back to 87%. The tiny gaps between top models lie within the 95% CI for n=958 — not statistically reliable.
🔩
Framework is the Bottleneck
The main finding is as much a framework finding as an architecture finding. PyTorch eager-mode on Linux/ARM adds per-layer overhead that scales with layer count. For production use, TFLite or ONNX Runtime (or CMSIS-NN on Cortex-M) are recommended in the Studienarbeit.
